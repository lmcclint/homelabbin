# Plex on the Core k3s Cluster — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Deploy Plex Media Server on the `fed1/2/3` k3s cluster, config-as-code via Ansible, with media from the UNAS Pro over NFSv3 (read-only), Plex config on Longhorn, QuickSync hardware transcoding, and LAN-only access on a MetalLB VIP.

**Architecture:** A single-replica `Deployment` in a new `plex` namespace (same `Recreate` + RWO-config pattern as the DNS apps). Media is a hand-written static NFS `PersistentVolume` bound 1:1 to a `ReadOnlyMany` PVC; config is a dynamic Longhorn PVC. A `LoadBalancer` Service (MetalLB VIP `10.20.20.201`, `externalTrafficPolicy: Local`) exposes port 32400. QuickSync is enabled by host-mounting `/dev/dri`. A new `plex` Ansible role applies the manifests and is wired into `site.yml` after `dns`; the `plex.lab.2bit.name` DNS record is seeded by the existing `dns` role.

**Tech Stack:** k3s, Ansible (`kubernetes.core.k8s`), MetalLB (L2), Longhorn, NFSv3, Technitium (DNS seeding), LinuxServer Plex image (`lscr.io/linuxserver/plex`).

**Reference spec:** `docs/superpowers/specs/2026-06-10-plex-k3s-design.md`

**Verified facts baked into this plan (do not re-derive):**
- `nfs-utils` is **already installed** by the `node-prep` role (Longhorn prereq) — no node-prep change needed.
- NFS media export `192.168.1.189:/var/nfs/shared/media` mounts only over **NFSv3** (v4 fails on the friendly path); allowed to node IPs `10.20.20.11/12/13`; `all_squash` means Plex reads it as **any** UID/GID.
- `render` group GID = **105** on all three nodes; `/dev/dri/renderD128` is mode **0666** (world-accessible).
- MetalLB `general-pool` is `10.20.20.200-254` (autoAssign). Gateway holds `.200`. We pin Plex to **`.201`** via annotation.

## File Structure

**Kubernetes manifests** (`kubernetes/plex/`, mirrors `kubernetes/dns/`):
- `namespace.yaml` — the `plex` Namespace.
- `media-pv.yaml` — static NFS `PersistentVolume` (cluster-scoped), bound to the media PVC via `claimRef`.
- `pvcs.yaml` — `plex-media` PVC (ReadOnlyMany, static) + `plex-config` PVC (RWO, Longhorn).
- `deployment.yaml` — the Plex `Deployment` (env, volumes, `/dev/dri`, securityContext).
- `service.yaml` — `LoadBalancer` Service, VIP `.201`, port 32400, `externalTrafficPolicy: Local`.

- `tasks/main.yml` — apply namespace → media PV → PVCs → claim secret (conditional) → deployment → service → wait for rollout.

## Claim-token handling (refinement of the spec)

The spec said store `PLEX_CLAIM` in ansible-vault. **This plan instead passes it as a deploy-time extra-var**, because the token is single-use and expires ~4 minutes after generation — a committed vault value would be stale on every later run. The Deployment references the Secret with `optional: true`, so:
- **First deploy:** generate a token at <https://plex.tv/claim>, then run with `-e plex_claim_token=claim-XXXXXXXX`. The role renders the Secret and Plex claims itself.
- **Every later run:** omit the var. The role skips the Secret; `optional: true` means the empty/absent env is harmless. Plex stays claimed via its `/config` volume.

## Task 2: Plex Deployment and Service (no HW transcode yet)

Get Plex running, claimed, and reachable on the LAN first. Hardware transcoding is layered on in Task 5 so each step is independently verifiable.

**Files:**
- Create: `kubernetes/plex/deployment.yaml`
- Create: `kubernetes/plex/service.yaml`

- [ ] **Step 1: Create the Deployment (software-only for now)**

`kubernetes/plex/deployment.yaml`:
```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: plex
  namespace: plex
spec:
  replicas: 1
  strategy:
    type: Recreate                 # single RWO config volume — never two pods at once
  selector:
    matchLabels:
      app: plex
  template:
    metadata:
      labels:
        app: plex
    spec:
      containers:
        - name: plex
          image: lscr.io/linuxserver/plex:latest   # pinned to a concrete tag in Task 6, Step 7
          env:
            - name: PUID
              value: "1000"
            - name: PGID
              value: "988"          # mirrors media group ownership (cosmetic; all_squash ignores it)
            - name: TZ
              value: America/New_York
            - name: VERSION
              value: docker          # pin Plex to the image's bundled version; no in-place self-update
            - name: PLEX_CLAIM
              valueFrom:
                secretKeyRef:
                  name: plex-claim
                  key: token
                  optional: true     # absent on re-runs after the first claim — harmless
          ports:
            - { containerPort: 32400, protocol: TCP }
          volumeMounts:
            - name: config
              mountPath: /config
            - name: media
              mountPath: /media
              readOnly: true
            - name: transcode
              mountPath: /transcode
          resources:
            requests:
              cpu: 500m
              memory: 1Gi
            limits:
              memory: 6Gi            # no CPU limit so transcodes can burst
      volumes:
        - name: config
          persistentVolumeClaim:
            claimName: plex-config
        - name: media
          persistentVolumeClaim:
            claimName: plex-media
        - name: transcode
          emptyDir: {}               # ephemeral transcode scratch; keeps churn off Longhorn
```

- [ ] **Step 2: Create the Service**

`kubernetes/plex/service.yaml`:
```yaml
apiVersion: v1
kind: Service
metadata:
  name: plex
  namespace: plex
  annotations:
    metallb.io/loadBalancerIPs: "10.20.20.201"
spec:
  type: LoadBalancer
  # Local preserves the real client IP (no kube-proxy SNAT) so Plex distinguishes
  # LAN vs remote clients correctly. With MetalLB L2, only the node running the pod
  # announces the VIP. Same lesson applied to Pi-hole.
  externalTrafficPolicy: Local
  selector:
    app: plex
  ports:
    - { name: pms, port: 32400, targetPort: 32400, protocol: TCP }
```

- [ ] **Step 4: SELinux contingency — only if Step 3 fails or transcode errors**

If `/dev/dri` is present on the host but the pod can't open `renderD128`, check the node for denials:
```bash
cd /home/lee/git/homelabbin/ansible
# find which node runs the pod, then inspect its audit log:
NODE=$(KUBECONFIG=/home/lee/git/homelabbin/kubeconfig kubectl -n plex get pod -l app=plex -o jsonpath='{.items[0].spec.nodeName}' | cut -d. -f1)
ansible "$NODE" -m shell -a 'ausearch -m AVC -ts recent 2>/dev/null | grep -i dri || echo NO_DRI_AVC'
```
If `dri` AVC denials appear, move the container into the `spc_t` (super-privileged container) domain, the standard fix for a container needing host device access under enforcing SELinux. In `deployment.yaml`, extend the pod `securityContext`:
```yaml
      securityContext:
        supplementalGroups: [105]
        seLinuxOptions:
          type: spc_t                # unconfined container domain — allows host device access
```
Re-apply and re-run Step 3. Note `spc_t` is permissive (the container runs largely unconfined); it's the pragmatic choice for a homelab. If you prefer least-privilege, the alternative is a small custom SELinux policy module allowing `container_t` → `dri_device_t` access, built from the captured AVCs with `audit2allow`. Document whichever approach you took in a comment on the `securityContext`.
